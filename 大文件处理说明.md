# 大文件处理说明

## v1.1 新功能：支持大文件翻译

### 问题说明

在 v1.0 中，虽然程序可以加载大文件，但由于 tkinter 的 Text 组件性能限制，一次性显示大量文本（超过10,000字符）可能导致：
- 界面卡顿
- 加载缓慢
- 滚动不流畅

### 解决方案

v1.1 引入了智能预览模式：

#### 1. 自动检测文件大小
- **小文件**（≤ 10,000 字符）：正常显示完整内容
- **大文件**（> 10,000 字符）：自动切换到预览模式

#### 2. 预览模式
- 只在界面上显示前 10,000 字符
- 完整文本仍然保存在内存中
- **翻译时使用完整文本**（不受显示限制影响）

#### 3. 手动切换
- 点击"显示完整原文"按钮查看全文
- 点击"仅显示预览"切换回预览模式

## 技术细节

### 实现原理

```python
# 1. 设置预览限制
self.preview_limit = 10000  # 字符数

# 2. 加载时判断文件大小
char_count = len(content)
is_large_file = char_count > self.preview_limit

# 3. 根据模式显示
if is_large_file and not self.show_full_text:
    # 只显示前 10,000 字符
    preview_text = self.current_text[:self.preview_limit]
else:
    # 显示完整文本
    display_text = self.current_text

# 4. 翻译时始终使用完整文本
segments = self.split_text_into_segments(self.current_text)
```

### 内存管理

- **完整文本**始终保存在 `self.current_text` 中
- **显示文本**是临时截取的片段
- 内存占用 = 完整文本大小（无额外开销）

### 翻译处理

```python
# 分段翻译（基于完整文本）
segments = self.split_text_into_segments(
    self.current_text,  # 使用完整文本
    max_length=500
)

# 示例：50,000 字符的文件
# 分段数 ≈ 50000 / 500 = 100 段
# 翻译时间 ≈ 100 段 × 1秒/段 = 100秒 ≈ 1.7分钟
```

## 常见问题

### Q1: 预览模式会影响翻译吗？

**A:** 不会！预览模式只影响显示，翻译时使用完整文本。

### Q2: 可以修改预览大小吗？

**A:** 可以，在代码中修改：
```python
self.preview_limit = 20000  # 改为20,000字符
```

### Q3: 为什么选择10,000字符作为阈值？

**A:** 经过测试，10,000字符是 tkinter Text 组件性能的平衡点：
- < 10,000: 流畅显示
- 10,000 - 50,000: 可用但略慢
- > 50,000: 明显卡顿

### Q4: 最大支持多大的文件？

**A:** 理论上无限制，实际取决于：
- **内存**：需要能容纳整个文本
- **API限制**：某些API有单次请求的token限制
- **翻译时间**：大文件需要更长时间

测试数据：
- 500K 字符 ≈ 1000 段 ≈ 16分钟翻译时间
- 1M 字符 ≈ 2000 段 ≈ 33分钟翻译时间

### Q5: 如何处理超大文件（>1M）？

**建议方案：**

1. **方案一：分章节翻译**
   - 手动分割成多个文件
   - 分别翻译后合并

2. **方案二：修改分段大小**
   ```python
   segments = self.split_text_into_segments(
       self.current_text,
       max_length=1000  # 增加到1000字符/段
   )
   ```
   - 减少API调用次数
   - 但单次翻译时间更长

3. **方案三：使用命令行批处理**
   - 适合自动化场景
   - 可以设置定时任务
